# Admin tooling for authoring trait_config.yaml (#39) — Design

**Issue:** Team-Hamsa/LFG #39 "Admin tooling for authoring trait_config.yaml
(config-gen UI)". Note: the spec/plan links already commented on #39 belong to
the **rules engine** (#40 body-affinity, `2026-07-04-trait-rules-body-affinity-*`).
This document is the first spec for the *tooling* itself.

## 1. What trait_config is NOW (verified, post #122-#126)

- **File:** `trait_config.yaml` at repo root, **548 lines**, ~20 comment lines
  (3-line header pointing at the rules-engine spec + section comments). The
  bulk (~500 lines) is the `affinity` map. `layers` / `z_overrides` /
  `swap_matrix` use compact **flow-style** entries
  (`- {name: Background, z: 10, shared: true}`); affinity values are quoted
  strings mapping to body lists (`"Balloon": [female, male]`).
- **Schema** (`lfg_core/trait_config.py`):
  - `version: 1` (required, trait_config.py:231-232)
  - `layers`: `[{name, z, shared?}]` → `LayerSpec` (:26-31, :234-242);
    duplicate names and empty list are load errors.
  - `z_overrides`: `[{trait_type, value, z}]` → `ZOverride` (:33-37, :244-249);
    unknown layer is a load error.
  - `affinity`: `{trait_type: {value: [bodies]}}` (:251-256); bodies must be in
    `VALID_BODIES = {ape, female, male, skeleton}` (:15).
  - `swap_matrix`: `universal_layers: [...]` + `pairs:
    [{bodies, layers | layers_except}]` → `SwapPair` (:40-44, :258-285);
    exactly one of `layers`/`layers_except` per pair.
  - `exclusions`: `[{trait_type, value, excludes: [{trait_type, values: "*"|[...]}]}]`
    — shape strictly validated (`_check_exclusions`, :177-225, including the
    bare-scalar-`values` substring-match footgun) but **the list is empty at
    launch**.
  - `inclusions`: scaffolded, **no consumer yet**; only "is a list" is
    validated (:293-299). **Out of scope for the editor** — do not offer
    authoring for a field nothing reads.
- **Validation entry points:**
  1. `load_config(path)` — structural (trait_config.py:228-309), raises
     `TraitConfigError`.
  2. `validate_against_store(cfg, store)` — async cross-check of affinity
     claims / layer names against the actual `layers/**` tree
     (trait_config.py:118-168); returns `(errors, warnings)`; gracefully
     degrades to structural-only when the (gitignored) layer tree is empty.
  3. `scripts/validate_trait_config.py` — CLI gate wrapping both, runs in
     pre-commit/pre-push and CI, exit 1 on errors.
- **How the engine consumes it:** lazily cached module-global singleton —
  `get_config()` loads on first call and caches in `_config`
  (trait_config.py:312-324). Consumers: mint selection
  (`lfg_core/traits.py:47`), swap compose (`lfg_core/swap_compose.py:23,59,91`),
  the service (`lfg_service/app.py:773,821`), economy API
  (`webapp/economy_api.py:185`). `reset_config()` exists (:322-324) but nothing
  in the running services ever calls it.
- **Existing satisfiability behavior:** `select_random_attributes`
  (lfg_core/traits.py:49-64) filters each layer's values through
  `value_allowed` + `conflicts` and **raises
  `ValueError("trait rules leave no legal <layer> value for body '<b>'")`**
  when rules eliminate every candidate. So the engine already fails loud on an
  over-constrained config — *at mint time*, in production. The editor's job is
  to move that failure to authoring time (§6).
- **Discord `/admin` today** (`surfaces/discord_bot/admin.py`): one `AdminView`
  (:386-469) with 6 buttons (stats, lookup, burn, view-odds, boost, toggle),
  each heavier op behind a `Modal` (max 5 text inputs each; selects cap at 25
  options). It has **rarity** controls but nothing touching trait_config.

## 2. Who edits this today (honesty section)

The user and dev sessions. The affinity map was generated by the PR-1 audit and
human-reviewed once (see the launch-slice comment on #39); ongoing edits are
occasional and surgical: add a z-override, add an exclusion, adjust one value's
body list, (rarely) insert a layer. There is **no community-admin audience
today**, and the file lives in git with pre-commit/CI validation — meaning the
real current workflow is "edit YAML, commit, restart", and the real current
pain is (a) hand-typing 548-line-file edits without typos, (b) forgetting the
restart, (c) not knowing whether an edit strands a body.

## 3. Surface choice — three options compared

| | 1. Discord /admin embed | 2. Activity web panel (drag/reorder, preview) | 3. Guided CLI (`scripts/trait_config_edit.py`) |
|---|---|---|---|
| Effort | Medium (modals per op; but writing files from the bot host, then needing a restart of the very process that took the command) | High (new admin auth surface, drag UI, live compose preview) | **Low** (pure script; reuses `load_config`/`validate_against_store` directly) |
| Audience fit | Future non-dev admins; overkill now | Same, plus best UX — but no such audience yet | **The actual current editors** (user + dev sessions) |
| Risk | Bot process writes a repo file it itself imported (stale cache, git dirty state on the deploy host, no diff review) | **Contradicts a same-day decision**: `2026-07-05-web-ui-rescope-design.md` §5 explicitly deferred browser admin — "a web session proves wallet ownership, not operator status, and shipping [admin] behind a new auth surface on a newly public host is the riskiest possible first tenant" | Lowest: git-native, diff-reviewable, validated before write, no auth surface at all |
| Discord constraint reality | Affinity (~500 entries) can't fit 25-option selects; only add-exclusion / set-z / small ops are modal-shaped | n/a | No constraint; full schema addressable |

**Decision: option 3 — guided CLI — is the MVP.** It matches the actual
audience, is consistent with the web-ui-rescope deferral (which this design
adopts rather than re-litigates), and delivers the safety properties (§4-6)
that are the *point* of #39 — "can't produce an invalid config" — without any
UI. Growth path: once a non-dev admin audience exists, wrap the same
edit-primitives module (§4) in a Discord `/admin` "Trait rules" sub-flow for
the modal-shaped ops only (add/remove exclusion, set z-override — NOT affinity
bulk edits); the web panel remains deferred behind the rescope spec's trigger
("re-open as its own issue if operating from Discord becomes painful") plus an
admin-auth design.

## 4. CLI design — `scripts/trait_config_edit.py`

Subcommands (non-interactive, scriptable, git-friendly), all sharing one
pipeline: **parse → mutate → validate → diff → (confirm) → backup → write**.

```bash
.venv/bin/python scripts/trait_config_edit.py [--config PATH] [--layers-dir layers] \
    [--dry-run] [--yes] <subcommand> ...

  set-z         --trait-type Eyes --value "Laser" --z 95      # add/update z_override
  del-z         --trait-type Eyes --value "Laser"
  set-affinity  --trait-type Accessory --value "Knife" --bodies male,female
  del-affinity  --trait-type Accessory --value "Knife"        # → dir-derived default
  add-exclusion --trait-type Head --value "Crown" \
                --excludes 'Accessory:*' --excludes 'Back:Wings,Cape'
  del-exclusion --trait-type Head --value "Crown"
  add-layer     --name Aura --z 85 [--shared]                 # prints TRAIT_ORDER warning (see below)
  reorder       --order Background,Back,Body,... [--accept-z-drift]
                # rewrites layer z as 10,20,...; z_override drift analysis below
  check                                                        # validate + satisfiability only, no edit
```

- **Editing primitives live in `lfg_core/trait_config_edit.py`** (pure
  functions over the ruamel document + a `Change` description), so a future
  Discord flow imports the same tested mutations; the script is argparse glue.
- **Default flow prints a unified diff and asks for confirmation**; `--yes`
  skips the prompt (for agents), `--dry-run` prints the diff and validation
  result and exits without writing. Exit codes: 0 written/clean, 1 validation
  failed, 2 satisfiability gate failed, 3 aborted at prompt.
- **`add-layer` caveat (verified):** `lfg_core/traits.py:48` — "Layers added
  only to trait_config.yaml won't mint until TRAIT_ORDER is updated too"
  (`lfg_core/swap_meta.py` `TRAIT_ORDER`; parity test
  `test_default_config_parity_with_legacy_constants` fails on divergence). The
  subcommand must print this as a blocking notice naming the constant and the
  test. The editor does NOT edit Python source.
- **`reorder` × `z_overrides` drift analysis (defined behavior):** overrides
  carry *absolute* z values (e.g. `Eyes/Wavy z=95` sits between old layers
  z=90/z=100); rewriting layer z to 10,20,… can silently move an override's
  *relative* position. After computing new layer z values, the tool locates
  every z_override in the old scale (which two layers it sits between, or
  above-all/below-all) and in the new scale, and prints a table:
  `override → old position (between X z=a / Y z=b) → new position (between
  A z=c / B z=d)`. If ANY override's relative position changed, `reorder`
  **hard-fails with exit 2** — nothing written — unless `--accept-z-drift` is
  passed (which writes and still prints the table). Overrides whose relative
  position is preserved are listed as `unchanged`. The tool does not attempt
  automatic re-mapping in MVP (an override's intent may be "above everything",
  not "between these two"); the table + explicit flag keeps the human the
  authority.
- **No `inclusions` subcommand** (no consumer, §1).

## 5. Round-trip safety

- **ruamel.yaml (new dev-time dependency) — justified.** The file has a header
  comment block, section comments, deliberate flow-style entries and quoting
  (§1). PyYAML `safe_dump` of the parsed tree would destroy all comments,
  explode flow style to block style, and re-quote keys — a 548-line diff for a
  one-line edit, unreviewable and history-hostile. ruamel round-trip mode
  (`YAML(typ="rt")`) preserves comments, flow style, quotes, and key order;
  edits produce minimal diffs. Cost: one pure-Python pip dependency used only
  by the editor module — the engine keeps loading with PyYAML `safe_load`
  unchanged (trait_config.py:230), so the runtime dependency surface does not
  grow. Pin in requirements.txt (`ruamel.yaml>=0.18`).
- **Validation before write, via the engine's own validator:** after mutating
  the ruamel doc, serialize to a temp file in the scratch dir, run
  `load_config(tmp)` and `asyncio.run(validate_against_store(cfg,
  LocalLayerStore(layers_dir)))` — exactly the gate
  `scripts/validate_trait_config.py` runs, so the editor can never write a
  config the pre-commit hook would reject. Any error → nothing written.
- **Backup:** before writing, copy the current file to
  `backups/trait_config/trait_config.<UTC-timestamp>.yaml`. Git is the primary
  safety net (the file is tracked); the backup covers dirty-tree edits.
  `backups/` is a **new path — nothing ignores it today**; the implementing PR
  adds `backups/` to `.gitignore`. **Retention:** keep the newest 20 backups;
  on each write, after copying, prune older ones (sorted-glob unlink loop over
  `backups/trait_config/trait_config.*.yaml`). Unbounded copies of a 548-line
  file would be harmless for years, but the bound is stated and tested rather
  than left implicit.
- **Dirty-file guard:** if `git status --porcelain trait_config.yaml` is
  non-empty, refuse unless `--force-dirty` — prevents silently stacking an
  edit on uncommitted changes (relevant given parallel Claude sessions; see
  concurrent-sessions memory).
- **Atomic write:** write temp file in the same directory + `os.replace`.
- **Idempotent semantics:** `set-*` upserts; `del-*` on a missing entry exits
  0 with "no change" (nothing written, no backup).

## 6. Guardrails — satisfiability gate

The engine already has the failure (`traits.py:58-64` raises on an
over-constrained layer), but only at mint time. The editor reuses the *same
code path* as an authoring-time smoke gate:

- **Deterministic exhaustive check (primary):** for each body ×
  `TRAIT_ORDER` layer, compute `[v for v in store.list_values(body, layer)
  if cfg.value_allowed(body, layer, v)]`; empty-with-raw-values → hard fail
  (mirrors traits.py:58 exactly, minus randomness). Cheap, complete for
  affinity edits.
- **Sampled draw gate (for exclusions, which are order/combination
  dependent):** run `select_random_attributes(store, body, rng=random.Random(seed),
  conn=sqlite3.connect(":memory:"))` **N=200 draws per body** (seeded, so failures
  reproduce); any `ValueError` → fail with the body/layer/seed printed.
  **The bare in-memory conn is verified safe:** `rarity.weighted_pick` calls
  `ensure_schema(conn)` and `_ensure_rows(...)` itself (lfg_core/rarity.py:153-154),
  creating the `trait_rarity` table and inserting floor-weight rows for every
  available value — an empty DB degrades to uniform floor weights, and
  `recalculate_rarity` explicitly guards for a missing `LFG` table
  (rarity.py:197) before querying it. No pre-seeding fixture is needed. This
  is probabilistic by nature — the spec is honest that a rare conflict chain
  can slip through; the mint path's own loud ValueError remains the backstop.
- Skipped with a warning when `layers/` is empty (same degradation as
  `validate_against_store`, trait_config.py:124-135) — e.g. CI; on dev/deploy
  machines with real art the gate always runs.
- `audit_layer_coverage.py` (on-chain coverage of *minted* NFTs) is a
  different question (chain vs config) and is **not** part of this gate; the
  editor's `check` subcommand prints a pointer to it for post-hoc audits.

## 7. Config reload (documented, not redesigned)

`get_config()` caches forever in a module global (trait_config.py:312-319);
no service calls `reset_config()`. Therefore **a written config takes effect
only after restarting the consuming processes**:

```bash
pm2 restart lfg-activity lfg-bot lfg-telegram
```

The editor prints exactly this after every successful write. **Hot reload is
explicitly not built now**: the config also feeds long-lived flows
(swap_compose mid-swap, mint sessions) and a mid-flow config swap has
unstudied consistency implications; a restart is a 2-second pm2 op the
post-merge hook already performs for lfg-activity (lfg-services memory).
Revisit only if edit frequency ever makes restarts painful.

## 8. Not building now (phase 2 triggers)

- **Drag-to-reorder / live compose-preview web editor** — deferred with the
  web-ui-rescope §5 admin deferral. Triggers to revisit: (a) a real non-dev
  admin audience exists, (b) a web admin auth design (operator status, not
  wallet ownership) is specced, (c) config edit frequency > ~weekly.
- **Discord `/admin` trait-rules sub-flow** — phase 2, not MVP; only the
  modal-shaped ops, importing `lfg_core/trait_config_edit.py` primitives.
  Trigger: the user wants to edit rules away from a dev machine.
- **Affinity bulk editing UI / CSV import-export** — the PR-1 audit already
  generated the bulk table; if a wholesale re-derivation is ever needed, that
  audit path is the tool, not this editor.
- **`inclusions` authoring** — blocked until a consumer PR defines the shape.
- **Hot reload** — §7.

## 9. Assumptions vs verified

- **Verified by reading code at the cited lines:** schema, validators, cache
  behavior, traits.py fail-loud, admin.py structure, yaml size/style,
  web-ui-rescope §5 deferral text (origin/main 6cd4a54), validate script env
  guard, TRAIT_ORDER coupling comment, and the sampled-gate rarity path
  (weighted_pick self-bootstraps schema/rows on any conn — rarity.py:153-154,
  :197).
- **Assumption:** ruamel.yaml round-trips this specific file's flow-style +
  quoting losslessly — plan task 1 verifies with a byte-diff test before any
  editing code is written; if it fails, fallback is targeted line-based edits
  for the flow-style sections (worse, but the test decides).
- **Assumption:** editors run on a machine with the `layers/` art tree
  (currently true for user + deploy host). The tool degrades to structural
  validation elsewhere, with a loud warning.
